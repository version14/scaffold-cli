> Status: not started.

## Goal

Make dot genuinely useful beyond a single Go REST API stub.

A developer should be able to scaffold any of the common project shapes — frontend, backend, monorepo, microservices — in any of the common languages, with the architecture pattern of their choice, and with a complete dev environment and CI/CD pipeline out of the box.

---

## Project types

| Type | Description |
|------|-------------|
| **Single project** | One app, one language. Existing. |
| **Monorepo** | Multiple apps in one repo, different languages. Shared CI. |
| **Microservices** | Gateway + services, auto-linked. Service communication via REST or gRPC. |

### Microservices specifics

A microservices project has a gateway (nginx / Kong / Traefik — chosen at `dot init`) and one or more services. When a new service is added via `dot add service`, it is automatically registered in the gateway config.

Optional: gRPC for service-to-service communication (instead of or alongside REST).

## Architecture patterns (chosen at dot init)

Selected at `dot init` time. Implemented via **generator composition**: each architecture pattern has its own standalone generator. API generators compose the selected architecture generator to produce the folder structure, then add their own language/framework-specific files on top.

```
dot init (go + rest-api + clean architecture)
  → GoRestAPIGenerator.Apply(spec)
      → GoCleanArchGenerator{}.Apply(spec)   ← folder structure
      → append Go REST API files (main.go, etc.)
  → pipeline.Run(merged ops) → project on disk
```

### Backend architecture generators

| Generator | Name | Pattern | Folder structure |
|-----------|------|---------|-----------------|
| `GoMVCGenerator` | `go-mvc` | MVC | `handlers/`, `models/`, `routes/` |
| `GoCleanArchGenerator` | `go-clean-arch` | Clean Architecture | `domain/`, `usecases/`, `interfaces/`, `infrastructure/` |
| `GoHexagonalGenerator` | `go-hexagonal` | Hexagonal | `core/` (domain + ports), `adapters/` (primary + secondary) |
| `TSMVCGenerator` | `ts-mvc` | MVC | `controllers/`, `models/`, `routes/` |
| `TSCleanArchGenerator` | `ts-clean-arch` | Clean Architecture | `domain/`, `usecases/`, `interfaces/`, `infrastructure/` |
| `TSHexagonalGenerator` | `ts-hexagonal` | Hexagonal | `core/`, `adapters/` |
| `PythonMVCGenerator` | `python-mvc` | MVC | `routers/`, `models/`, `schemas/` |
| `PythonCleanArchGenerator` | `python-clean-arch` | Clean Architecture | `domain/`, `usecases/`, `interfaces/`, `infrastructure/` |

Each architecture generator is registered in the registry and can be used standalone. API generators compose them — they are not exclusive to their parent generator.

### Frontend architecture generators

| Generator | Name | Pattern | Folder structure |
|-----------|------|---------|-----------------|
| `TSFeatureSlicedGenerator` | `ts-feature-sliced` | Feature-sliced Design | `features/`, `entities/`, `shared/`, `pages/`, `app/` |
| `TSAtomicDesignGenerator` | `ts-atomic-design` | Atomic Design | `atoms/`, `molecules/`, `organisms/`, `templates/`, `pages/` |
| `TSContainerPresentationalGenerator` | `ts-container-presentational` | Container/Presentational | `containers/`, `components/`, `pages/` |

Frontend generators (React, Next.js, Vue) compose these in the same way API generators compose backend architecture generators.

## Dev environment

### Docker Compose (dev only)

Generates a `docker-compose.yml` for local development. Includes all declared modules (Postgres, Redis, etc.) as services. The app itself is not in the compose file — developers run it locally.

This is dev environment only. Production deployment is handled separately (see Deployment section below).

A `docker-compose.override.yml` can extend the base for local customisation.

## Deployment

All deployment artifacts are optional modules selected at `dot init`. Multiple can be combined.

| Module | What it generates |
|--------|------------------|
| `github-actions-deploy` | Deploy workflow for the selected cloud provider (ECR push, ECS deploy, etc.) |
| `terraform-aws` | Terraform modules for AWS (VPC, ECS, RDS, etc.) |
| `terraform-gcp` | Terraform modules for GCP (Cloud Run, Cloud SQL, etc.) |
| `kubernetes` | Kubernetes manifests + Helm chart |

Cloud provider is selected at `dot init` (aws / gcp). The deployment modules generate provider-specific files accordingly.

CI is GitHub Actions only in v0.2. GitLab CI and others are planned for v0.6.

## Add-on tools

Modules added at `dot init` or later via `dot add module`. Each generates configuration, client setup code, and integrates with the CI pipeline.

### Monitoring and observability
- `grafana` — Grafana dashboard config + data source setup
- `sentry` — Sentry SDK init + error boundary (frontend) / middleware (backend)
- `posthog` — PostHog analytics client setup

### Frontend tools
- `tanstack-router` — TanStack Router setup for React/Vue
- `tanstack-query` — TanStack Query (React Query) client setup
- `shadcn-ui` — shadcn/ui component library setup (Next.js / React)
- `payload` — Payload CMS setup for Next.js

### Backend tools
- `grpc` — protobuf setup + gRPC server/client stubs
- `graphql` — GraphQL schema + resolver scaffold (Go: gqlgen, Node: Apollo)

## CI — GitHub Actions

The existing CI workflow (vet, lint, test, build) is extended dynamically.

When a module is added that requires a CI service (e.g. Postgres), the CI workflow is updated to add the corresponding service container.

When a deployment module is added (e.g. `terraform-aws`), a deploy workflow is generated or updated.

CI provider: GitHub Actions only in v0.2. Planned: GitLab CI in v0.6.

## Custom generators (local)

Users can write a generator in Go, point dot at it, and use it immediately — no publishing required.

```bash
dot generator add ./my-generator    # register a local generator
dot generator list                  # show registered generators
dot generator remove my-generator   # unregister
```

Local generators implement the same `Generator` interface as official ones. They are stored in `.dot/generators.json`.
